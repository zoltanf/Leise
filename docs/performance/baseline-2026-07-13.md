# Leise Performance Baseline — 2026-07-13

Status: accepted baseline
Architecture checkpoint: two fixed runtime bundles and `LeisePluginSDK` remain

This report closes Milestone A of the remaining simplification and performance
plan. It is the comparison point for composition, startup, memory, build, and
transcription work. Compact raw data is archived under
[`data/baseline-2026-07-13`](data/baseline-2026-07-13/).

## Methodology

- Source: commit `4208303e36ee4bd6a0cefbc99308ffe4f8e2f38e`, branch
  `leise-foundation`, with an intentionally dirty migration worktree. The
  captured source-tree fingerprint is
  `efd01b9a3eb8656912732b0aa2044ea05305c60f35bb7cbaf5534574e31063a3`;
  the exact file inventory and dirty-state listing are in the
  [environment archive](data/baseline-2026-07-13/environment/).
- Machine: 10-core Apple M1 Pro MacBook Pro with 32 GB RAM, macOS 26.5.2,
  arm64, Xcode 26.6 (17F113). Measurements ran on AC power with the battery at
  80% and no concurrent Leise process. See
  [capture notes](data/baseline-2026-07-13/environment/capture-notes.md).
- Launch and transcription runs used the Debug app. Build and artifact results
  used both Debug and Release with code signing disabled and a dedicated
  `.build/DerivedData-Performance` directory.
- Launch, first-use transcription, and latency summaries contain seven measured
  runs. Builds contain three runs because clean Release builds take roughly 109
  seconds each. Median and p90 use linear interpolation over the ordered values.
- Warm launch used completed onboarding, disabled update and feedback-sound
  work, and no loaded model. Cold launch is deliberately omitted because no
  repeatable OS cache condition was established.
- The downloaded Parakeet v2 and v3 models were measured after one excluded
  Core ML compiler-cache warm-up. “First use” therefore means a fresh app
  process with the model downloaded but not loaded, not a never-compiled Core ML
  cache.
- Fixtures are locally generated with macOS voice `Samantha` at rate 180, then
  converted to 16 kHz mono PCM. Their source text, durations, and SHA-256 hashes
  are archived in the [fixture manifest](data/baseline-2026-07-13/fixtures/fixtures.csv).
  Regeneration produced identical audio hashes on this machine.
- The fixture runner disables vocabulary boosting and measures model
  preparation, local transcription, and normalization. It does not include
  microphone capture, post-processing persistence, or Accessibility insertion,
  so this report makes no microphone-to-insertion latency claim.
- Every run in a given fixture scenario produced identical text. The short
  fixture matched its source text exactly; medium and long output differed only
  in deterministic punctuation from their source text.

## Known variance and interpretation

- The v3 first-use set contains one 613.262 ms run; this raises p90 while the
  394.708 ms median remains close to six of seven runs. Both statistics remain
  visible rather than deleting the outlier.
- Release clean-build p90 is descriptive only with three samples. Later build
  comparisons must keep the same run count, destination, package state, and
  DerivedData procedure.
- macOS scheduling, background services, filesystem cache state, and thermal
  history were not laboratory-isolated. Foreground workloads were minimized,
  the machine was on AC power, and `pmset` did not provide a useful thermal
  pressure reading.
- Synthetic speech removes microphone and room variance but is not a proxy for
  recognition accuracy across real speakers. These fixtures are performance
  regression inputs only.
- Loaded-model memory varies with fixture length and transient inference
  buffers. The independent short first-use samples are the accepted v2/v3
  settled-memory comparison; longer single-process curves remain diagnostic.

## Accepted budgets

The provisional budgets are accepted and made concrete against the median
values above:

- warm UI readiness at or below 223.731 ms and hotkey readiness at or below
  209.394 ms (at least 30% lower);
- pre-model idle RSS at or below 73,248 KiB (at least 25% lower);
- Debug clean at or below 13,972.731 ms, Release clean at or below 81,519.195
  ms, and Debug incremental at or below 4,046.529 ms;
- no named launch main-thread interval above 100 ms without a documented
  platform constraint—the current 115.833 ms container-construction median is
  a demonstrated bottleneck;
- no more than 5% median regression in first-use v3 (414.443 ms), first-use v2
  (518.415 ms), steady short (173.181 ms), steady medium (449.568 ms), or steady
  long (1,036.669 ms) fixture transcription.

P90 remains a required reported diagnostic, but the median is the budget gate.
Any final comparison must use the same fixture hashes, model-cache condition,
run counts, build configurations, and measurement scripts. Milestone B may now
begin; no cold-launch or microphone-to-insertion improvement may be claimed from
this baseline.
