# Leise final performance report — 2026-07-13

Status: implementation accepted with documented startup, memory, and build exceptions

Architecture checkpoint: direct static components through `LeiseComponents`

This report closes the measurement portion of the remaining simplification and
performance plan. Compact raw data is archived under
[`data/final-2026-07-13`](data/final-2026-07-13/); the comparison baseline is
[`baseline-2026-07-13.md`](baseline-2026-07-13.md).

## Methodology

- Same machine and configuration as baseline: 10-core Apple M1 Pro, 32 GB RAM,
  macOS 26.5.2, Xcode 26.6, arm64, on AC power.
- Same commit and intentionally dirty migration worktree. The final captured
  source fingerprint is
  `d8f46638a50c7eaaf88e3b018109ba548c6db62e855e3b57cabe0b5386201987`.
- Seven warm launches and seven measured transcription runs per scenario.
  Builds retain the baseline's three-run count.
- Fixture hashes exactly match the baseline. Each scenario produced identical
  transcript text on every run.
- A one-time 31-second Core ML compiler-cache warm-up appeared after rebuilding
  the static component. As required by the baseline methodology, that warm-up
  was excluded and the seven-run v3 first-use set was repeated after the cache
  was warm.

## Launch and memory

Times are milliseconds and memory values are KiB. Change is based on medians;

The ordering objective succeeded: hotkey monitoring is ready before retained
store work and post-launch maintenance. The broader UI and idle-memory budgets
did not. Direct static construction and the still-eager dictation correctness
stores leave the composition root as the largest startup interval. This is the
primary remaining runtime bottleneck.

That tradeoff is accepted for this architecture checkpoint; dependency build
optimization remains separate future work.

## Verification and exceptions

- `swift test --package-path LeiseComponents --scratch-path .build/LeiseComponents`:
  34 XCTest cases and 2 Swift Testing cases passed.
- Full `Leise` macOS test target passed with `xcodebuild test` on arm64.
- Clean Debug and Release builds succeeded in every measured run.
- `scripts/check_static_components.sh` passes and is part of PR preflight.
- The built app has no `Contents/PlugIns`; retained Parakeet and filler-cleanup
  identifiers are present in the app binary.

A separate clean Git clone was not created because this checkpoint is an
uncommitted repository migration; clean DerivedData builds verified resolution
and compilation of the captured source tree. The manual permission, real
microphone, native-app/browser insertion, offline-download, and cancellation
matrix remains a release smoke-test gate and is not represented as automated
evidence here.

## Decision

The static architecture, correctness suites, structural guard, hotkey ordering,
and dictation budgets are accepted. UI startup, idle memory, and build-time
targets are explicit misses. Further work should begin with a profile of
composition-root construction and FluidAudio compilation, not with restoring
runtime bundles or speculative hot-path changes.
